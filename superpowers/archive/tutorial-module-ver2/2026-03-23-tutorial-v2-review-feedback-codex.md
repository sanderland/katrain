# Tutorial Module V2 规划审核反馈

整体判断：这份规划的优点是分解细、文件落点明确、实现路径可执行，不是“空话计划”。但它同时存在几处关键缺口，尤其是导入阶段对真实 `book.json` 的文本建模不完整、初始 `board_payload` 来源未落地、字母编辑逻辑与需求不一致、viewport 半盘裁剪方案实际上未闭环，以及 JSON→DB 迁移的回滚/兼容策略不足。如果不先修这些点，实施中途很容易出现“代码写了很多，但需求核心没真正落地”的情况。

---

### 1. 需求 (1) 整页截屏
**判定：** ✅
**说明：**
- 计划明确把左侧资源改成 `page_image_path`，并在页面里展示整页截图，不再依赖切割后的单独棋图。
- `TutorialFigure.page_image_path` 的设计可以天然支持“多个变化图共享同一页截屏”。这一点与需求一致。
- 我抽样看了真实数据：`/Users/fan/Repositories/go-topic-collections/books/布局/曹薰铉布局技巧_上册_曹薰铉_1997/output/book.json` 第 11 页确实有两个 `figure_ref`，共享同一页截图是实际存在的场景，计划模型能承载。
**建议（如有）：**
- 如果后续想减少重复字段，可把“页面级资源”再抽成 `tutorial_pages` 表；但这不是当前必须项。

### 2. 需求 (2.1) 摆子功能（黑/白/交替 + 计数）
**判定：** ⚠️
**说明：**
- `useBoardEditor` 和 `BoardEditToolbar` 已覆盖黑/白/交替三种模式，以及编号开关，方向上是对的。
- 但“交替”目前靠 `B/W` 总数判断下一手颜色，这不稳定。已有初始局面、手工擦除、或中途切换模式时，计数法不一定等价于“下一次点击真正应该轮到谁”。
- 编号逻辑只在 `numbering=true` 时递增，基本符合需求，但没有说明取消编号再打开后的连续性是否按历史最大编号继续。
**建议（如有）：**
- 把“交替”实现成独立的 `nextStoneColor` 状态，而不是靠当前黑白子数量推断。
- 进入编辑模式时同时推导 `nextStoneColor` 和 `moveCounter` 的初值，并在成功落子后再推进。
- 给这一块补单元测试，覆盖“已有局面 + 擦除 + 切换模式”。

### 3. 需求 (2.2) 字母编辑键的累加逻辑
**判定：** ❌
**说明：**
- 这是目前最明确的功能性错误。计划里的 `handleClick` 把已有字母的位置视为 `isOccupied`，而字母工具只有在 `!isOccupied` 时才会执行。
- 需求明确要求“在同一位置连续点击多次也会累加”，含义是：同一位置允许反复点，盘面显示最后一次字母，但中间字母位必须被消耗。
- 按现在的写法，同一位置第二次点击不会消耗字母位，也不会更新为下一个字母，和需求相反。
**建议（如有）：**
- 对 `activeTool === 'letter'` 单独处理：
  - 若该点已有字母且没有石子/图形，允许覆盖成下一个字母并推进计数。
  - 若该点有石子或图形，仍然禁止。
- 进入编辑模式时，从现有最大字母继续；保存后重新进入，也必须从持久化后的最大字母继续。
- 增加专门测试：同一点连点三次后应显示 `C`，并确认 `A/B` 已被消耗。

### 4. 需求 (2.3) 图形编辑键
**判定：** ✅
**说明：**
- 计划覆盖了图形下拉选择、三种图形类型、以及点击空位落图形，核心需求是覆盖到的。
- 渲染方案也与当前 `SGFBoard` 的 SVG 结构匹配，实现可行。
**建议（如有）：**
- 明确“同一点已有图形时再次选择另一种图形点击，是否替换”。需求没写，但实现前最好定掉。

### 5. 需求 (2.4) 保存 + 模式切换 + viewport 裁剪
**判定：** ⚠️
**说明：**
- “编辑模式显示完整 19×19 棋盘，保存后退出编辑模式”这一层是覆盖了。
- 但只读态裁剪并没有真正闭环。计划的 `compute_viewport` 在半盘场景上是明显不完整的，后面第 13 条会详细说。
- 另外，计划同时写了 “Toolbar 内有 Save” 和 “页面上有 save/cancel buttons”，交互边界不清晰，容易造成重复实现。
- 更大的问题是取消编辑逻辑没有定义“撤销未保存修改”。当前 hook 只在初次 `useState` 时取 `initialPayload`，如果用户编辑后点取消，再次进入编辑模式，很可能看到的是上次未保存的内存状态，而不是数据库中的已保存状态。
**建议（如有）：**
- 明确“编辑态的工作副本”和“已保存 payload”分离。
- `cancel` 必须重置为最后一次保存成功后的 payload。
- `save` 入口只保留一个，避免 toolbar 与 page 同时持有保存职责。

### 6. 互斥约束（同一位置不可同时有落子/字母/图形）
**判定：** ⚠️
**说明：**
- 计划的方向是对的，确实尝试在 `handleClick` 里用 `hasStone / hasLetter / hasShape` 做互斥。
- 但它把“已有字母”与“禁止再次点击字母”混在了一起，和第 3 条的需求冲突。
- 另外，`hasStone/hasLetter/hasShape` 是用外层 `payload` 先算，再进入 `setPayload(prev => ...)`；这比直接基于 `prev` 判断更脆弱，容易留下状态不同步隐患。
**建议（如有）：**
- 互斥规则应写成更细的矩阵：
  - 石子不能和字母/图形共存。
  - 图形不能和石子/字母共存。
  - 字母不能和石子/图形共存，但允许覆盖已有字母并消耗计数。
- 占用判断放进函数式 updater 内，统一基于 `prev` 计算。

### 7. 擦除功能
**判定：** ✅
**说明：**
- 计划中的 eraser 会删除黑白子、编号、字母、图形和 highlight，覆盖面是够的。
- 就“能否擦除三种元素”这个问题，答案是可以。
**建议（如有）：**
- 需求同时提了“擦除/撤销”，当前只有 eraser，没有 undo。这个问题应单独补上，不要把 eraser 当作 undo 的替代。

### 8. 多步棋回放（move-step slider）
**判定：** ⚠️
**说明：**
- 计划注意到了这个需求，并且提出从 `labels` 推导最大步数，这个方向基本合理。
- 但它只说“显示 label <= currentStep 的石子”，没有定义未编号石子的处理方式。真实盘面可能存在“基础局面 + 编号变化图”混合场景，未编号石子通常应一直可见。
- 如果用户在编辑中关闭编号再摆子，后续回放怎么解释，也没有说清楚。
**建议（如有）：**
- 明确回放语义：未编号石子是“初始静态局面”，始终显示；带编号石子按步进显隐。
- 只对满足“编号序列连续且为数字”的 payload 显示 slider。
- 给 API/前端都加测试：有未编号基础局面时，步进不应把基础石子隐藏掉。

### 9. 数据库表结构是否完整覆盖左侧/右侧内容
**判定：** ⚠️
**说明：**
- 从字段名看，`page_image_path + book_text + board_payload + narration + audio_asset` 基本覆盖了需求描述中的左右内容。
- 但这里有两个关键缺口：
  - 计划里导入时把 `board_payload` 设成 `None`，而需求要求初始展示 visual LLM 识别结果。
  - 导入时只取 `figure_ref.text`，真实 `book.json` 里大量页面还含有独立的 `description` 元素，且很多 `figure_ref.text` 本身是空的。
- 我抽样统计了当前书库，约有 10,140 个 `figure_ref`，其中 1,560 个 `text` 为空。这意味着只存 `figure_ref.text` 会直接丢失大量原文。
**建议（如有）：**
- 明确初始 `board_payload` 的来源：来自 review 产物、单独识别结果文件，还是现有 Phase 1 JSON。这个来源必须落地进导入脚本。
- 文本建议至少拆成两层：
  - `page_context_text` 或 `page_description`
  - `figure_text`
- 如果短期不想加字段，也要定义清楚合成规则，而不是直接 `elem.get("text", "")`。

### 10. CLI 导入脚本是否处理 Type A / Type B
**判定：** ⚠️
**说明：**
- 这份计划按“每章判断是否有 `sections`”来分支，而不是按整本书判断，这一点是对的。因为真实数据里确实存在“同一本书大部分章节没 sections，但个别章节有 sections”的混合情况。
- 我抽样看了 `/Users/fan/Repositories/go-topic-collections/books/入门/曹薰铉和李昌镐围棋教室_入门篇_上_2000/output/book.json`，16 章里有 15 章没有 sections；这说明“按章判断”比“按书判断”稳。
- 但导入脚本没有任何测试任务，且文本合成、初始 `board_payload`、`--force` 资产覆盖都没解决，所以不能判定为完整通过。
**建议（如有）：**
- 把 `tests/web_ui/test_tutorial_import.py` 从文件结构里的“挂名文件”落成真实任务。
- 至少覆盖 4 类用例：
  - 纯 Type A
  - 纯 Type B
  - 同书混合章结构
  - 同页多个 `figure_ref`

### 11. 数据库 schema（4 张表）设计是否合理
**判定：** ⚠️
**说明：**
- 层级拆分成 `book / chapter / section / figure` 是合理的，也符合页面导航结构。
- 级联删除方向基本正确，`order` 字段也有必要。
- 但 schema 还不够“生产可用”：
  - 没有 parent 内部的顺序唯一约束，例如 `(book_id, order)`、`(chapter_id, order)`、`(section_id, order)`。
  - 没有分类约束，`category` 只能靠应用层保证。
  - `board_payload` 更新没有版本字段或乐观锁支持。
  - `page_image_path` 按需求应是导入后的必备资产，长期看不应一直允许 `NULL`。
**建议（如有）：**
- 增加复合唯一约束和必要索引。
- 对 `category` 使用 `CheckConstraint` 或枚举。
- 为 `TutorialFigure` 增加 `version` 或直接使用 `updated_at` 做并发校验。

### 12. 一级分类 hardcoded 而不是 DB 表
**判定：** ⚠️
**说明：**
- 如果这四个一级分类在中期内是稳定固定的，hardcoded 可以接受，实施成本也更低。
- 风险在于：
  - 导入脚本、后端查询、前端显示三处都要同步维护。
  - 没有 referential integrity，手误输入分类时数据库不会拦。
  - 未来扩展二级分类或重命名时，迁移会更痛。
**建议（如有）：**
- 短期保留 hardcoded 没问题，但至少在 DB 层加 `CheckConstraint`。
- 如果后面确定会扩展“棋书”之外的二级内容，建议尽早引入 `tutorial_categories` 表。

### 13. `compute_viewport` 的 1/4、1/2 棋盘裁剪逻辑
**判定：** ❌
**说明：**
- 这是第二个明确的功能设计问题。
- 计划里自己已经承认“当前 `SGFBoard` 是正方形 viewport，需要矩形支持”，但 `compute_viewport` 仍然在 top/bottom half 场景返回 `size=10`。这实际上是 10×10，不是半盘。
- 也就是说，需求里的“1/2 棋盘”在当前计划里并没有被真正实现，只是被注释说明了一下。
- 此外，算法只按象限分布判断，缺少更严格的“所有占用点都落在 snapped viewport 中”的验证。
**建议（如有）：**
- 明确采用矩形 viewport：
  - 左半盘：`{ col: 0, row: 0, cols: 10, rows: 19 }`
  - 右半盘：`{ col: 9, row: 0, cols: 10, rows: 19 }`
  - 上半盘：`{ col: 0, row: 0, cols: 19, rows: 10 }`
  - 下半盘：`{ col: 0, row: 9, cols: 19, rows: 10 }`
- 先写测试，再写算法，至少覆盖四角、四个半盘、跨对角线必须退回全盘、边界星位包含等场景。

### 14. `SGFBoard` 从只读扩展为可编辑
**判定：** ⚠️
**说明：**
- 基于现有 SVG 棋盘加 `onClick` 和 invisible rects，是一条务实路线，复用度高。
- 但计划文档内部自己也不一致：
  - 文件结构里有 `EditableSGFBoard.tsx`
  - 任务里却只修改 `SGFBoard.tsx`
  - 页面描述又写“右侧使用 `EditableSGFBoard`”
- 这说明组件边界还没有定下来。
**建议（如有）：**
- 建议保留 `SGFBoard` 为纯渲染组件，再用 `EditableSGFBoard` 做一层交互包装：
  - 读写职责分离
  - 只读模式不会被编辑逻辑污染
  - 与文档中的文件结构保持一致

### 15. `useBoardEditor` 用 `JSON.parse(JSON.stringify(prev))` 深拷贝
**判定：** ⚠️
**说明：**
- 以 19×19 盘面的数据量来说，性能未必会立刻爆炸，但这仍然是脆弱实现。
- 它会隐式丢掉 `undefined`、未来可能加进来的复杂值，也会让状态更新路径不够可控。
- 更重要的是，当前 `handleClick` 的占用判断是在 updater 外层做的，这本来就该重构。
**建议（如有）：**
- 用 `structuredClone` 或手写小范围不可变更新。
- 更好的做法是把每个工具的更新逻辑抽成纯函数，并为这些纯函数写测试。

### 16. API 设计（RESTful endpoints）是否合理
**判定：** ⚠️
**说明：**
- 路由层级和前端页面层级大体对得上，命名也比较直观。
- 但数据粒度设计还有改进空间：
  - `/books/{id}` 只返回 chapters，不返回 sections，导致 `TutorialBookDetailPage` 需要按 chapter 再发一轮请求，形成 N+1。
  - `PUT /figures/{id}/board` 接受任意 `Dict[str, Any]`，验证太弱。
  - 计划没有考虑谁有权限修改教程内容，这对一个共享内容库来说不是小问题。
**建议（如有）：**
- 提供一个面向页面的聚合接口，例如 `/books/{id}/tree`，直接返回 chapter + section 树。
- 给 `board_payload` 建立严格的请求模型，而不是裸 `dict`。
- 修改接口至少应有认证和角色控制。

### 17. `PUT /figures/{id}/board` 是否需要考虑并发编辑冲突
**判定：** ❌
**说明：**
- 计划完全没有处理并发覆盖问题。
- 当前设计是“客户端拿到旧 payload，编辑后整包 `PUT` 回去”，这意味着后保存的人会无提示覆盖先保存的人。
- 在教程内容是共享编辑资产的前提下，这会直接造成数据丢失。
**建议（如有）：**
- 至少做乐观锁：
  - 返回 `updated_at` 或 `version`
  - `PUT` 时要求客户端回传
  - 不匹配则 409
- 如果短期不做多人协作，也至少在计划里明确“当前只允许单管理员离线编辑”，不要默默忽略。

### 18. 资产路径安全：`_safe_asset_path` 是否足够
**判定：** ⚠️
**说明：**
- 它比直接拼字符串安全得多，但仍然不够严谨。
- `str(resolved).startswith(str(base))` 是经典易踩坑写法，理论上存在前缀伪装问题，应该用路径级比较而不是字符串前缀比较。
- 另外，路径值来自数据库导入，也应该在导入时做一次合法性限制，不要把风险全部留给读取接口。
**建议（如有）：**
- Python 3.9+ 直接用 `resolved.is_relative_to(base)`。
- 或用 `os.path.commonpath([resolved, base]) == str(base)`。
- 对 `page_image_path` 的导入值限制在 `tutorial_assets/{slug}/pages/` 之下。

### 19. 6 个 Chunk 的顺序是否合理，能否并行
**判定：** ⚠️
**说明：**
- 大方向顺序是合理的：先数据层，再导入，再 API，再前端，再清理。
- 但有两个问题：
  - 破坏性删除出现得太早，见第 21 条。
  - Chunk 4 和 Chunk 5 在 API contract 稳定后是可以部分并行的，计划没有利用这一点。
**建议（如有）：**
- 建议顺序改成：
  - Chunk 1 DB schema
  - Chunk 2 import + import tests
  - Chunk 3 API + API tests
  - Chunk 4/5 前端页面与编辑器并行推进
  - Chunk 6 仅保留 cleanup，放到所有验证通过之后

### 20. 从 JSON 迁移到 DB 的过渡期风险与回滚方案
**判定：** ❌
**说明：**
- 计划写了“DB is sole source of truth; JSON files deleted”，但没有给任何平滑迁移方案。
- 当前代码里 tutorial loader 是在 `server.py` 启动时挂到 `app.state` 上的，现有系统是能工作的。直接切掉 loader，意味着一旦导入不完整，整个教程模块会立刻失效。
- 也没有看到“导入后做 count parity / sample parity / shadow read”的步骤。
**建议（如有）：**
- 加 feature flag，例如 `TUTORIALS_V2_ENABLED`。
- 先做 shadow import 和对账：
  - 分类数
  - 书/章/节/图数量
  - 抽样资产可访问
  - 抽样 `board_payload` 非空率
- 确认 V2 可用后，再切默认读取路径。
- 保留回滚开关，至少一版内不要直接删除旧 JSON 数据。

### 21. 删除旧文件的时机是否安全
**判定：** ❌
**说明：**
- 不安全。计划在 Chunk 3 就删除 `loader.py` / `progress.py`，在 Chunk 4 删除旧页面，在 Chunk 6 删除旧数据目录。
- 这些删除都发生在“新系统经过完整端到端验证”之前。
- 一旦中间某个点卡住，仓库会同时处于“旧路径被删、新路径未完全可用”的不稳定状态。
**建议（如有）：**
- 所有 destructive cleanup 都放到最后一个 commit。
- 在 cleanup 前必须完成：
  - 后端测试通过
  - 前端 build 通过
  - 至少一轮 Playwright 覆盖编辑保存成功
  - 真实导入样本验证通过

### 22. 测试覆盖是否充分
**判定：** ❌
**说明：**
- 当前测试计划覆盖面不够，尤其对“最危险的逻辑”覆盖不足。
- 主要缺口包括：
  - `test_tutorial_import.py` 在文件结构里出现，但正文没有任何任务。
  - 没有 `useBoardEditor` 单元测试。
  - 没有 viewport 半盘场景测试。
  - 没有路径遍历测试。
  - E2E 只验证“能进编辑态并点保存”，没有真正验证摆子/字母/图形/擦除/持久化。
- 这意味着最容易出错的部分几乎都没被自动化保护住。
**建议（如有）：**
- 补三层测试：
  - 后端：import / viewport / PUT board auth+validation / 409 conflict
  - 前端单元：hook 和 `SGFBoard` 渲染
  - E2E：真实点击棋盘、保存、刷新后仍然存在

### 23. 是否有需求明确要求但计划遗漏的功能
**判定：** ❌
**说明：**
- 有，且不止一个：
  - 初始展示 visual LLM 识别结果：计划里导入直接把 `board_payload=None`，没有说明识别结果从哪里来。
  - 撤销功能：需求写了“需要擦除/撤销功能”，计划只有 eraser，没有 undo。
  - 同一位置字母连续点击累加：第 3 条已证明没实现。
- 这些都不是锦上添花，而是需求正文里的明确项。
**建议（如有）：**
- 把“初始 `board_payload` 来源”单独列成一个任务和数据迁移策略。
- 在 `useBoardEditor` 中增加 undo stack。
- 为字母覆盖累加写专门测试后再实现。

### 24. 是否有计划中出现但需求未提及的功能（过度设计）
**判定：** ⚠️
**说明：**
- 没有看到特别严重的“大型过度设计”，大多数扩展都还是围绕需求展开。
- 但有几处复杂度略超需求：
  - 额外拆出多条层级 API，再在前端做多轮请求，复杂度高于“一个聚合树接口”。
  - 文档里出现 `EditableSGFBoard.tsx`，但实现任务又回到 `SGFBoard.tsx`，说明组件拆分尚未收敛。
  - `subcategory="棋书"` 现在只是占位字段，如果短期不会用到，不必在前后端都大量展开其存在感。
**建议（如有）：**
- 对“未来可能用到”的结构保留字段即可，不要让当前实现承担过多前瞻复杂度。

### 25. `UserTutorialProgress` 表如何处理
**判定：** ❌
**说明：**
- 这是规划里一个明显的未决项。
- 现有代码里 `UserTutorialProgress` 仍在使用，当前 API 也暴露了 `/progress` 读写接口。
- 新计划删除了 `progress.py`，但没有说明：
  - 进度功能是下线还是保留
  - 如果保留，旧 `example_id/topic_id` 怎么映射到新 `section/figure`
  - 旧测试如何迁移
**建议（如有）：**
- 必须显式定一条策略：
  - 保留进度：则设计新 progress schema 或映射层
  - 暂时下线：则删表、删接口、删前端引用、删测试，一次删干净
- 不要保留一个“半死不活”的 progress 数据模型。

### 26. `slugify` 处理中文书名是否可靠
**判定：** ⚠️
**说明：**
- 目前这版 `slugify` 并不会转成 ASCII，而是保留中文字符。它能工作，但和注释里“URL-safe slug”的说法并不一致。
- 好消息是，我扫了当前 20 本书的目录名，按这个算法没有出现 slug 冲突。
- 风险仍然存在：
  - 未来若目录名清洗后变空，会回退成 `"book"`，直接冲突。
  - 部署、备份、跨平台脚本若默认假设 ASCII 路径，可能踩坑。
**建议（如有）：**
- 如果 slug 只用于文件路径，保留 Unicode 也可以，但注释要改准确。
- 更稳妥的是：
  - 基于目录名生成 slug
  - 再附加短 hash 或递增后缀
  - 保证唯一性由代码而不是人工假设来承担

### 27. 前端 loading / error / empty 状态
**判定：** ❌
**说明：**
- 计划没有系统性提到这些状态。
- 现有页面虽然很多地方已有 `loading` spinner 的模式，但新规划没有把这些状态列为实现要求，也没有对应测试。
- 对教程这种多层级异步页面，缺这些状态会直接影响可用性：
  - 分类无书
  - 书无章节
  - section 无 figure
  - 资产 404
  - 保存失败
**建议（如有）：**
- 在每个新页面都明确三类状态：
  - loading
  - error with retry
  - empty
- 编辑保存失败时要有 toast/snackbar，而不是静默失败。

### 28. 19×19 之外的棋盘是否需要支持
**判定：** ⚠️
**说明：**
- 当前需求正文只明确了 19×19 编辑态，所以“不支持 9×9/13×13”不是 bug。
- 但计划一方面把 `BoardPayload.size` 设计成泛化字段，另一方面 `compute_viewport` 又直接对非 19 返回 `None`，属于“接口看起来支持、核心逻辑又没真支持”的半泛化状态。
**建议（如有）：**
- 这期建议明确写成非目标：
  - 编辑功能只保证 19×19
  - 非 19×19 只读全盘显示
- 同时在 API 验证里把这个约束写清楚，避免未来调用方误判。

### 29. 其他值得指出的问题或改进建议
**判定：** ❌
**说明：**
- 还有几处计划里没被点名、但实际很重要的问题：
  - `PUT /figures/{id}/board` 没有认证/授权控制。教程内容是共享资产，匿名用户不应能直接改库。
  - `board_payload` 请求体没有强校验，非法结构会直接入库。
  - 导入脚本先 `session.commit()`，再复制图片；如果复制失败，数据库里会留下指向不存在资产的记录。
  - `--force` 重导入时没有清理旧资产目录，可能留下脏文件。
  - 计划把 `data/tutorial_assets/` 一起 `git add`，这很可能把大量截图资产塞进主仓库，仓库体积和 review 成本都会飙升。
**建议（如有）：**
- 把编辑接口至少限制为管理员或内部工具调用。
- 给 `board_payload` 建严格 schema 校验。
- 导入流程改成“先复制并验证资产，再提交事务”，或至少在失败时清理半成品。
- 明确 `tutorial_assets` 是否入 git；如果只是运行时资产，应该加入忽略规则，不要默认 commit。

---

## 总结

**整体评分：** 5/10

**主要优点：**
- 结构化程度高，Chunk / Task / 文件路径 / 命令都写得很清楚，可执行性强。
- 数据模型方向是对的，`Book → Chapter → Section → Figure` 能很好承载新教程层级。
- 前后端大体沿着同一份 contract 在推进，不是完全脱节的两套方案。

**必须修复的问题：**
- 明确初始 `board_payload` 的真实来源，不能导入成 `None` 后再“以后再说”。
- 修正字母工具同点连续点击的逻辑。
- 重写 half-board / quarter-board 的 viewport 设计与测试。
- 补上 undo、cancel reset、并发冲突处理。
- 明确 `UserTutorialProgress` 的去留与迁移方案。
- 为编辑接口增加 auth / validation。
- 把 import / editor / persistence 的测试补齐后再谈删除旧系统。

**建议优化的问题：**
- 为 schema 增加复合唯一约束、分类约束、版本字段。
- 重新建模书页文本，至少区分页级描述和 figure 说明。
- 优化 slug 生成策略与资产目录策略。
- 给所有新页面补 loading / error / empty 状态。
- 把 cleanup 作为最后一步，而不是实现中途就开始删旧路径。
