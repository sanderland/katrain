## 原始需求

### 背景

KaTrain 是一个围棋教学应用，已完成教程模块第一版（JSON 文件驱动、只读 SVG 棋盘）。第二版需要解决以下问题并新增功能。

### 需求 (1)：左侧图片改为整页截屏

- 放弃使用 visual LLM 切割后的棋局图片，改为展示棋局所在的**整个页面截屏**。
- 每本书的页面截屏已存在于 `go-topic-collections/books/{分类}/{书名}/output/pages/page_NNN.png`。
- 一页中可能包含多个棋局（figure_ref），多个变化图可以共享同一张页面截屏。

### 需求 (2)：右侧 SGF 棋盘改为可编辑模式

初始展示 visual LLM 识别的结果，点击"编辑"按钮可进入编辑模式：

**(2.1) 摆子功能**
- 三种模式：摆黑、摆白、交替。
- 计数开关：打开时摆子带从 1 开始的编号（落子序号）。

**(2.2) 字母编辑键**
- 图标为"ABC"，从 A 开始，每点击一次累加一个字母位。
- 即使在同一位置连续点击多次，也会累加（当前位置显示最后一次的字母，之前的字母位被跳过/消耗）。
- 最多到 Z，再点击弹窗提示已到头。
- 切换其他按钮再切回时，从盘面上已有的最大字母继续计数。

**(2.3) 图形编辑键**
- 下拉菜单选择图形（正方形、三角形、圆形等）。
- 选定后在盘面空位添加图形标记。

**(2.4) 保存与模式切换**
- 保存按钮：点击后退出编辑模式，变回只读模式。
- 编辑模式：展示整张 19×19 棋盘。
- 只读模式：只展示有子的最小棋盘局部——1/4 棋盘、1/2 棋盘、或整张棋盘。1/4 和 1/2 棋盘的边界要包含星位点。

**约束**
- 同一位置不可同时有落子、字母、图形（互斥）。
- 本期只涉及 SGF 编辑功能，AI 生成的文字和音频先不改动。
- 需要擦除/撤销功能，可以删除已放置的落子、字母或图形。

### 需求 (3)：数据库存储

- 新增数据库表存储教程模块所有棋局数据。
- 每个棋局包括：
  - 左侧：原始页面截图路径 + 原始书籍文字
  - 右侧：SGF board_payload（可编辑）+ AI 改编文字 + 音频
- 每次 SGF 编辑保存后，更新后端数据库的 board_payload 字段。
- 数据库是唯一数据源，删除本地 JSON 文件。

### 页面层级结构

```
一级分类：入门 / 布局 / 中盘 / 官子
二级分类：目前统一为"棋书"（后续按细分领域扩展）
三级分类：书名
四级及以下：章节 → 小节(=例子) → 变化图(=图1, 图2...)
```

- 对于没有 sections 的书（Type A，如入门篇），用章节标题生成合成 section。
- 每个页面展示一个变化图。
- 如果变化图涉及多步棋（有编号的落子），需要左右按钮和滑动条展示不同步的状态。
- "形势判断"归入"中盘"分类。

### 数据导入

- 使用 CLI 脚本从 `book.json` 导入数据到数据库。
- 页面截屏复制到 `data/tutorial_assets/{book_slug}/pages/`。

### board_payload 格式

```json
{
  "size": 19,
  "stones": {"B": [[2,16]], "W": [[3,3]]},
  "labels": {"2,16": "1", "3,3": "2"},
  "letters": {"5,5": "A", "6,6": "B"},
  "shapes": {"7,7": "triangle", "8,8": "square", "9,9": "circle"},
  "highlights": [],
  "viewport": {"col": 0, "row": 0, "size": 10}
}
```

## 审核要求

请逐条回答以下问题，给出"通过 ✅ / 有问题 ❌ / 建议改进 ⚠️"的判定，并附上理由。

### 一、需求覆盖度

1. 需求 (1) 整页截屏：计划是否完整覆盖？是否处理了"多个变化图共享同一页截屏"的情况？
2. 需求 (2.1) 摆子功能（黑/白/交替 + 计数）：`useBoardEditor` hook 和 `BoardEditToolbar` 是否完整实现？
3. 需求 (2.2) 字母编辑键的累加逻辑：计划中 `letterCounterRef` 的行为是否匹配"在同一位置连续点击多次也会累加"的需求？注意：需求说的是*在同一位置*点击多次，每次都累加字母——这意味着同一位置的字母会被覆盖为新字母，但中间的字母位被消耗了。
4. 需求 (2.3) 图形编辑键：下拉菜单选择 + 放置，是否完整？
5. 需求 (2.4) 保存 + 模式切换 + viewport 裁剪：是否完整？
6. 互斥约束（同一位置不可同时有落子/字母/图形）：是否在 `handleClick` 中正确实现？
7. 擦除功能：是否覆盖？能否擦除所有三种元素（落子、字母、图形）？
8. 多步棋回放（move-step slider）：是否在计划中覆盖？逻辑是否合理？
9. 数据库表结构是否完整覆盖了左侧（截图+文字）和右侧（board_payload+narration+audio）？
10. CLI 导入脚本是否处理了 Type A（无 sections）和 Type B（有 sections）两种书籍结构？

### 二、架构与技术设计

11. 数据库 schema（4 张表）的设计是否合理？外键关系、级联删除、索引是否恰当？

13. `compute_viewport` 的 1/4、1/2 棋盘裁剪逻辑是否正确？边界星位点是否被包含？
14. `SGFBoard` 从只读 SVG 扩展为可编辑：通过添加 `onClick` prop + 覆盖 invisible rects 的方式，是否合理？有没有更好的方案？

16. API 设计（RESTful endpoints）是否合理？路由层级是否与前端路由对齐？
17. `PUT /figures/{id}/board` 只更新 board_payload——是否需要考虑并发编辑冲突？
