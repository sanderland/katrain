# Tutorial Module V2 — Plan Review Feedback (Revision 2)

在查看了更新后的计划 `2026-03-23-tutorial-v2-plan.md` 后，我对照第一版的审核意见进行了复核。Claude 对所有的 ❌ 和 ⚠️ 项都给出了高质量的解决方案。

### 1. 需求 (1) 整页截屏覆盖度
**判定：** ✅
**说明：** 完美满足，逻辑清晰。新增了 `page_context_text` 用于关联描述，考虑更加全面。

### 2. 需求 (2.1) 摆子功能（黑/白/交替 + 计数）
**判定：** ✅
**说明：** 交替模式改进为使用显式的 `nextStoneColor` 状态（落子后翻转），避免了单纯依赖黑白子数量推断造成的逻辑漏洞。

### 3. 需求 (2.2) 字母编辑键的累加逻辑
**判定：** ✅ (前一版为 ❌)
**说明：** 计划在 `useBoardEditor.ts` 细化了互斥逻辑：允许同位置的新字母覆盖旧字母，并且在 `setActiveTool('letter')` 时重新扫描当前盘面以计算最大字母的 ASCII 码。彻底解决了之前无法覆盖和切回工具后标号错乱的问题。

### 4. 需求 (2.3) 图形编辑键
**判定：** ✅
**说明：** 新增了允许直接替换已有形状的支持，逻辑完整。

### 5. 需求 (2.4) 保存、模式切换与视图裁剪
**判定：** ✅

### 6. 互斥约束
**判定：** ✅ (前一版为 ⚠️)
**说明：** “Fine-grained mutual exclusion” 逻辑非常清晰，同类元素覆盖，异类元素拦截。所有的占用检查都安全地放在了 `setPayload(prev => ...)` 的更新函数内部，避免了闭包状态过期问题。

### 7. 擦除与撤销功能
**判定：** ✅
**说明：** 新增了基于 client-side 数组的 50 步 Undo Stack，并详细说明了橡皮擦清空各种标记的逻辑，完美匹配需求。

### 8. 多步棋回放（move-step slider）
**判定：** ✅
**说明：** 明确了“无编号的棋子视为初始局面始终显示，有编号的参与 slider 过滤”的语义，非常优雅解耦的设计。

### 9-11. 数据库结构与导入
**判定：** ✅
**说明：** 增加了 `CheckConstraint` 和 `UniqueConstraint` 保证数据一致性。导入脚本中要求“先复制图片再提交数据库”，避免了数据库存在而图片丢失的幽灵数据。

### 12. Hardcoded list 存 Category
**判定：** ✅ (前一版为 ⚠️)
**说明：** 在 `models_db.py` 中增加了 `CheckConstraint("category IN ('入门', '布局', '中盘', '官子')")`，在数据库层级保证了数据一致性，对于 V2 这是一个务实可靠的方案。

### 13. `compute_viewport` 裁剪逻辑与星位点
**判定：** ✅ (前一版为 ❌)
**说明：** Top Half 和 Bottom Half 已经修正为返回矩形视图（如 `cols: 19, rows: 10`），并修改了前端 `SGFBoard` 以支持非正方形裁剪。同时补充了对应的详尽边界测试，完美解决了半个棋盘丢失数据的 Bug。

### 14. `SGFBoard` 扩展可编辑
**判定：** ✅
**说明：** 这是一个非常棒的改进：将组件拆分为 `SGFBoard` (纯渲染) 和 `EditableSGFBoard` (交互包裹层)。不仅逻辑解耦，而且完美解决了我在上一版指出的“在移动端可能拦截滚动操作”的问题（非编辑模式下不挂载隐形点击网格）。

### 15. `useBoardEditor` 深拷贝性能
**判定：** ✅ (前一版为 ⚠️)
**说明：** 改用原生 `structuredClone(prev)` 替代 `JSON.parse(JSON.stringify)`，性能更好且符合现代 JS/TS 标准。

### 16. API 设计与并发编辑冲突
**判定：** ✅
**说明：** 引入了 `expected_updated_at` 字段实现乐观锁 (Optimistic locking)，遇到冲突返回 HTTP 409。同时给编辑接口加上了用户认证 (Auth) 和严谨的 Pydantic 校验，考虑得非常周全企业级。

### 17. 迁移过渡期风险与删除时机
**判定：** ✅ (前一版为 ⚠️)
**说明：** 引入了 `TUTORIALS_V2_ENABLED` Feature Flag，新旧双轨并行。并将所有破坏性的文件/数据删除动作推迟到了最后一个 Chunk（在所有测试验证通过之后执行），迁移风险降到了最低。

### 18. `UserTutorialProgress` 迁移问题
**判定：** ✅ (前一版为 ❌)
**说明：** 明确了废弃旧进度系统但保留表结构 (Task 15b)，去除了不再兼容的 API 接口，避免了迁移期的报错和潜在的数据丢失。

### 19. 前端 Loading / Error 状态
**判定：** ✅ (前一版为 ⚠️)
**说明：** 在 `TutorialFigurePage` 等页面明确定义了三态处理 (Loading spinner / Error retry / Empty state)，前端体验得到保障。
