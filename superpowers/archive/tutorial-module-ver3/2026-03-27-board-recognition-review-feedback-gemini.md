# Tutorial Module Stage 3 — Board Recognition Design Review Feedback

作为资深计算机视觉工程师和围棋软件架构师，我对当前的实施计划进行了全面的审核。总体而言，"CV 精确定位 + VLLM/小模型 分类" 的两阶段演进架构是非常务实且高效的。特别是利用 Contact Sheet 降低大模型调用成本、提高准确率的设计，体现了很好的工程权衡。

以下是逐项的评估与改进建议：

### 一、架构合理性

1. **"CV 定位 + VLLM 分类"的职责分离是否合理？**
   - **判定：通过 ✅ / 建议改进 ⚠️**
   - **理由**：非常合理。CV 在处理规则网格和精确定位时具备计算效率高、精度高的优势；而 VLLM 在处理复杂分类、抗干扰（如数字提取、特殊标记识别）上泛化能力强。
   - **改进建议**：不要完全依赖 VLLM。部分显而易见的无数字纯黑白子可以由 CV 脚本基于阈值直接给出高置信度分类，只有那些有数字、有标记、置信度低于某个阈值的 patch 才放进 Contact Sheet 交给 VLLM 处理，这样可以进一步压缩 VLLM 调用的 token 开销并提升处理速度。

2. **Contact sheet 方式批量发给 VLLM 是否最优？**
   - **判定：通过 ✅**
   - **理由**：是最优解。大模型在处理单图多零散物体时往往存在幻觉（比如多数一个子、少数一个子），但如果有序拼接并打上明确的网格标签（如 A1, A2...），VLLM 的回答会非常结构化。相比并发请求几十张小图，这种方式大幅节省了 Token 且成功率极高。

3. **CV 宽松检测 + VLLM 过滤 false positive：**
   - **判定：通过 ✅**
   - **理由**：符合 Recall（召回率）优先的策略。围棋谱识别中漏检的代价极高（少个子棋理就不通了），而误检（把空交叉点的污渍当成有子）交由 VLLM 纠正为 `empty` 的成本极低，逻辑严密。

4. **星位校准方案：**
   - **判定：有问题 ❌ / 建议改进 ⚠️**
   - **理由**：存在明显的隐患。如果是定式图或激烈的局部战斗，星位很可能完全被棋子覆盖，导致无法提取星位特征来进行坐标系锚定。
   - **改进建议**：星位匹配只能作为辅助手段。主要的坐标锚定应依赖于**棋盘物理边缘检测**（检测粗黑线或者外围留白）以及**落子坐标范围投影**。另外，可以将带边框的完整棋谱截图直接发给 VLLM，让其辅助推断当前视口对应的绝对坐标系（如：左上角，行 0-10，列 0-10）。

### 二、CV 技术方案

5. **gap-fill 的触发条件 (gap > 1.6× spacing)：**
   - **判定：通过 ✅**
   - **理由**：标准的网格距是 1x，1.6x 能够有效区分正常的连续网格和被棋子遮挡导致的缺失网格。只要边缘检测逻辑和内部网格逻辑分离，基本不会误触发。

6. **Occupied 检测的多维特征阈值泛化能力：**
   - **判定：有问题 ❌**
   - **理由**：硬编码的全局绝对阈值（如 `dark_ratio > 0.45`）在跨书泛化时必然失败，因为不同出版社的印刷深浅、扫描件的曝光度和对比度差异巨大。
   - **改进建议**：采用**自适应阈值**。可以在每张页面的网格区域内采样多个随机空交叉点的像素分布，动态计算基准 `mean` 和 `std`，将固定阈值替换为基于该页面基准的相对变化阈值（如 z-score > 3）。

7. **裁剪尺寸 50×50 px 是否足够：**
   - **判定：建议改进 ⚠️**
   - **理由**：取决于原图分辨率对应的实际网格像素。考虑到网格距约为 39px，50x50 基本覆盖了单个交叉点及其紧邻区域。但在包含两位数（如 88, 99）时，边缘可能会由于插值或压缩变得模糊。
   - **改进建议**：生成 Contact Sheet 前，对 patch 执行轻度的对比度拉伸（CLAHE）或锐化（Sharpening），去除贯穿数字的网格线干扰，以提升 VLLM OCR 的准确度。

8. **棋图 bbox 的 CV 投影分析：**
   - **判定：通过 ✅**
   - **理由**：排版规整的书籍使用 1D 水平/垂直投影找段落间隙是非常经典且高效的做法。如果偶尔失败，可以结合形态学膨胀操作将内容连通后求轮廓（Contours），鲁棒性已经足够。

### 三、VLLM 分类方案

9. **分类类别设计是否完备：**
   - **判定：有问题 ❌**
   - **理由**：不完备。围棋书中除了三角和字母，极其常见的标记还包括：**方形（Square/□）**、**叉形（Cross/X）**、**圆形（Circle/○）**（空心圆通常标在黑子上）、以及**半透明/虚线子**（表示假设的变化图）。此外，字母标记可能落在空交叉点上，也可能直接印在黑白子上。
   - **改进建议**：扩展类别体系，建议输出格式解耦：`{"color": "B|W|E", "text": "1-99|A-Z|null", "shape": "triangle|square|circle|cross|null"}`。

10. **VLLM 通过 subagent 调用的效率：**
    - **判定：建议改进 ⚠️**
    - **理由**：按单图 10-20 秒计，一本书需耗时几个小时。这作为离线异步脚本是可以接受的，但在开发和调试阶段会令人抓狂。
    - **改进建议**：放弃纯串行的 subagent，在 `process_book.py` 脚本中使用 `asyncio` + 大模型直接的 API 请求并发处理多个 Contact Sheet，能将时间缩短到几分钟内。

11. **数字 OCR 准确度：**
    - **判定：通过 ✅**
    - **理由**：当代顶级 VLLM（如 Claude 3.5 Sonnet / GPT-4o）对于拼图中的 40x40 规则印刷体数字有极高的识别率，基本不需要额外的 Tesseract，前提是图像够清晰。

### 四、Phase 2 模型训练

12. **EfficientNet-B0 的选择：**
    - **判定：通过 ✅**
    - **理由**：极其合适。针对此类分类任务，它的参数量小、特征提取能力强，完全可以做到端侧毫秒级推理。也可作为替代选项考察更轻量的 MobileNetV3。

13. **数据需求估算与不平衡：**
    - **判定：建议改进 ⚠️**
    - **理由**：2000 个 patch 够启动，但 `empty` 类别和纯黑白子将占据 90% 以上，而 `square`、`X` 或某些两位数标签会非常稀疏。
    - **改进建议**：训练时必须引入 Focal Loss 应对类别不平衡，并对少数类别执行数据增强（Data Augmentation：轻微旋转、亮度抖动）。

14. **双头设计（分类头 + 数字头）：**
    - **判定：通过 ✅**
    - **理由**：Multi-task Learning 在此类场景表现通常很好，强迫模型提取更通用的特征。可以设定一个共享的特征提取骨干网络，后接独立的全连接层。

15. **防止 VLLM 错误传播：**
    - **判定：极其重要 ⚠️**
    - **理由**：Garbage in, garbage out。VLLM 标注存在幻觉风险。
    - **改进建议**：在将 Phase 1 积累的数据用于 Phase 2 训练前，必须经过人工审核机制。可以编写一个简单的本地 HTML/JS 页面，一次性平铺展示 500 个同类别（如所有识别为 "white+N" 的图）的 patch 及其标签，人眼快速扫过挑出错误的即可。

### 五、工程实现

16. **单文件脚本是否臃肿：**
    - **判定：有问题 ❌**
    - **理由**：如果将图片读取、CV预处理、VLLM API封装、拼图逻辑、数据库写入全塞在 `recognize_boards_v2.py` 中，很快会超过 1000 行，变成技术债。
    - **改进建议**：强烈建议拆分为独立模块：
      - `core/vision/grid_detector.py`
      - `core/vision/patch_classifier.py`
      - `core/vision/contact_sheet.py`
      - `scripts/import_book_vision.py`

17. **自动保存训练数据的设计：**
    - **判定：建议改进 ⚠️**
    - **理由**：只按目录分类保存小图不利于后续调试和分析。
    - **改进建议**：除了保存图片，需要生成对应的 `metadata.jsonl`，记录诸如 `{ "patch_id": "xx", "source_page": "page_05", "source_figure": 1, "col": 4, "row": 3, "label": "B+5" }` 等信息，方便追溯原始上下文。

18. **错误处理策略：**
    - **判定：建议改进 ⚠️**
    - **理由**：批处理不应因为单张图片识别出错而整体崩溃。
    - **改进建议**：如果 VLLM 识别格式崩了或超时，Catch exception 并在 DB 中将该图标记为 `status="recognition_failed"`，然后继续处理下一张。

19. **端到端流程：**
    - **判定：通过 ✅**
    - **理由**：流程清晰，通过统一脚本串联是良好的实践。

### 六、与 Stage 2 的集成

20. **乐观锁冲突 (409)：**
    - **判定：通过 ✅**
    - **理由**：这是后台离线导入脚本，通常是在书籍发布到前端之前运行，不存在用户并发修改的情况。为了健壮性，脚本里加上简单的 retry 机制即可。

21. **Highlights 字段为空：**
    - **判定：通过 ✅**
    - **理由**：静态书籍的棋谱图通常通过字母或形状（三角/方块）来表示关注点，直接映射到 `shapes`/`letters` 是正确的。交互式的高亮（Highlights）应留给前端逻辑处理。

22. **用户修改反哺训练集：**
    - **判定：强烈建议 ✅**
    - **理由**：这是闭环系统最重要的一环。
    - **实现建议**：在后端的 Update Figure API 中加一段逻辑：如果请求修改了 `board_payload`（与原有的非 null 版本对比有 diff），则将变更（原坐标对应的截图、原标签、用户纠正的新标签）写入一张 `recognition_corrections` 修正表。积累一段时间后直接转化为高质量的 Hard Examples 用于微调 Phase 2 模型。

---

### 总结意见

整体实施计划**批准推进**。建议近期实施 Phase 1 时，优先解决**模块化拆分**、**支持多形状（方块/叉形/半透明）标记**以及**放弃绝对阈值改用自适应阈值**这三个核心痛点。星位校准可以先跑跑看，如果有问题，立刻切向基于边框或 VLLM 辅助的锚定策略。
